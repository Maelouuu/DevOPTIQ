Architecture Documenté

Vue d'ensemble

DevOPTIQ est une application web monopage classique (Flask + Jinja2). Pas de SPA : chaque page est rendue côté serveur. Le JavaScript vanilla gère les interactions riches (éditeur cartographie, appels API JSON asynchrones).

Flux d'une requête HTTP

  1. Le navigateur envoie une requête HTTPS vers Cloud Run.
  2. Gunicorn dispatche vers le worker Flask disponible.
  3. Flask résout la route → blueprint concerné → fonction de vue.
  4. La vue interroge SQLAlchemy (PostgreSQL en prod, SQLite en dev), rend le template Jinja2.
  5. La réponse HTML/JSON est renvoyée au navigateur.
  6. Pour les opérations IA (chatbot, import, propositions), la vue appelle l'API Anthropic via HTTP avant de répondre.

Isolation des données par entité

Toute l'application est multi-tenant au niveau de l'entité (organisation). Chaque utilisateur peut posséder plusieurs entités. L'entité active est stockée dans la session Flask (session['active_entity_id']), pas en base. Tous les modèles exposent for_active_entity() pour filtrer automatiquement.

Règle : ne jamais requêter Model.query.all() directement dans les vues. Toujours passer par Model.for_active_entity().all() ou filtrer explicitement par entity_id.

Stack technique Documenté

CoucheTechnologieNotes
Runtime Python 3.12 Image Docker python:3.12-slim. LibreOffice installé pour conversion de fichiers bureautiques.
Framework web Flask Pattern application factory (create_app()). Blueprints pour chaque domaine. Deux filtres Jinja2 custom : extract_numeric_level et escapejs.
WSGI Gunicorn Serveur de production. Lancé via startup.sh. Le port est injecté par Cloud Run via la variable $PORT.
Base de données PostgreSQL (prod) · SQLite (dev) Prod : PostgreSQL sur Neon (free tier → max ~20 connexions). Pool : pool_size=2, max_overflow=3, pool_recycle=300s.
Dev : SQLite WAL mode activé via PRAGMA, timeout 30 s, check_same_thread=False.
Variable d'environnement : DATABASE_URL (préfixe postgres:// corrigé en postgresql://).
ORM / Migrations Flask-SQLAlchemy · Flask-Migrate (Alembic) Les migrations Alembic ne sont pas exécutées au démarrage (risque de verrou DDL PostgreSQL → worker timeout). À la place, db.create_all() + ALTER TABLE … ADD COLUMN idempotents garantissent la cohérence.
Frontend HTML Jinja2 · CSS vanilla · JS vanilla Aucun framework JS ni bundler. $(sel) est un alias document.querySelector(sel) défini dans chaque page. CSS organisé par domaine (optiq.css = global). Icônes via Font Awesome CDN.
IA SDK openai → modèles Anthropic Claude Le client OpenAI est configuré avec le base URL et la clé API Anthropic. Utilisé pour : chatbot intégré, import IA global, propositions de compétences/softskills/savoirs/aptitudes. Clé : variable ANTHROPIC_KEY dans Cloud Run.
Cartographie OptiqCarto (outil maison) Éditeur SVG interactif. Voir section Cartographie.
Import fichiers vsdx · python-docx · openpyxl Import de diagrammes Visio (.vsdx), documents Word et classeurs Excel. Export Excel via openpyxl.
Email Flask-Mail · Gmail SMTP TLS 587 Utilisé pour le reset de mot de passe et les notifications. Compte afdec.enterprise.services@gmail.com. Credentials dans variables Cloud Run.
Auth Sessions Flask · bcrypt Login/logout classique avec sessions côté serveur. Pas de JWT. session['user_id'] et session['active_entity_id'] sont les deux clés centrales.
Déploiement Google Cloud Run · Docker Image construite depuis Dockerfile, déployée sur Cloud Run. Variables d'environnement sensibles configurées dans Cloud Run (pas dans le code) : DATABASE_URL, ANTHROPIC_KEY, SECRET_KEY, MAIL_PASSWORD.

Saisie de temps pour une activité, une tâche ou un rôle. Calcule automatiquement le temps annuel via des propriétés.

ColonneValeurs possibles

Journal d'audit automatique

Les modèles Activities, Task, Role et Tool sont instrumentés via des SQLAlchemy event listeners (after_insert, before_update, after_update, after_delete). Chaque mutation insère une ligne dans recent_events dans la même transaction, avec le diff avant/après pour les updates.

La fonction _capture_changes(target, fields) utilise l'historique SQLAlchemy (state.attrs[field].history) pour capturer les valeurs avant modification, sans requête supplémentaire.
