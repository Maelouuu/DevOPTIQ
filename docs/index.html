Architecture Documenté

DevOPTIQ est une application web monolithique Flask déployée sur Google Cloud Run. Toute la logique métier est côté serveur ; le frontend est rendu par Jinja2 et enrichi par du JavaScript vanilla sans framework.

Vue d'ensemble

Flux de données d'une requête HTTP

  1. Le navigateur envoie une requête HTTPS vers Cloud Run.
  2. Flask route la requête vers le blueprint correspondant (ex: activities_map_bp).
  3. Le blueprint interroge la base via SQLAlchemy, filtre par entité active (Entity.get_active()).
  4. Jinja2 rend le template HTML avec les données, renvoyé au navigateur.
  5. Pour les appels API (JS fetch), la réponse est du JSON.

Multi-entités et isolation des données

Chaque utilisateur possède une ou plusieurs entités (organisations). L'entité active est stockée dans la session Flask (session['active_entity_id']). Tous les modèles principaux exposent une méthode de classe for_active_entity() qui filtre automatiquement les données.

Cloud Run stateless : le filesystem est éphémère. Les SVG de cartographie et les fichiers uploadés sont stockés à la fois sur disque (cache court-terme) et en base de données (svg_content, optiqcarto_data, file_blobs) pour survivre aux redémarrages.

Stack technique Documenté

CoucheTechnologieRôle
Backend Python 3 · Flask Serveur web, routing, logique métier. Application factory pattern (create_app()). ~45 blueprints enregistrés au démarrage.
Base de données PostgreSQL (prod) / SQLite (local) PostgreSQL via Neon en production (pool limité à 2+3 connexions pour le free tier). SQLite avec WAL mode en développement local (Code/instance/optiq.db).
ORM SQLAlchemy via Flask-SQLAlchemy Mapping objet-relationnel. Migrations gérées par Flask-Migrate (Alembic), mais les colonnes manquantes sont ajoutées via ALTER TABLE au démarrage pour éviter les timeouts Cloud Run.
Frontend HTML Jinja2 · CSS vanilla · JS vanilla Pas de framework JS (React, Vue…). $() est un alias local de document.querySelector. CSS par domaine fonctionnel + optiq.css global.
IA Anthropic Claude API Utilisée pour 3 cas : chatbot contextuel, import IA (extraction de données depuis texte libre), propositions automatiques de compétences/savoirs/softskills.
Cartographie OptiqCarto (outil maison SVG) Éditeur SVG interactif développé en JS vanilla (editor.js). Importe des fichiers Visio (.vsdx). Synchronisé vers les modèles DB à chaque sauvegarde.
Mail Flask-Mail · Gmail SMTP Envoi d'emails pour reset de mot de passe et notifications. CustomMail corrige le local_hostname SMTP.
Auth Sessions Flask · bcrypt Authentification par email/mot de passe. Pas de JWT. L'utilisateur connecté est stocké dans session['user_id'].
Déploiement Google Cloud Run · Docker Image Docker construite et déployée sur Cloud Run. Variables sensibles (DB_URL, ANTHROPIC_KEY, SECRET_KEY…) injectées via l'interface Cloud Run.

Démarrage de l'application

create_app() dans app.py exécute ces étapes dans l'ordre :

  1. Charge les variables d'environnement (dotenv).
  2. Configure SQLAlchemy (PostgreSQL prod vs SQLite local, pool de connexions).
  3. Configure Flask-Mail (SMTP Gmail).
  4. Enregistre les ~45 blueprints.
  5. Dans le contexte applicatif : db.create_all() (idempotent), puis ALTER TABLE pour les colonnes manquantes, puis seed des données de démonstration si vide.
  6. Dispose le pool de connexions pour repartir proprement avant les requêtes HTTP.
Filtre Jinja2 personnalisé : extract_numeric_level extrait le premier chiffre d'une chaîne de niveau (ex: "Niveau 2" → "2"). escapejs échappe une chaîne pour injection sécurisée dans du JS inline.

Modèles principaux

Entity — entities

Représente une organisation (entreprise, département). C'est le pivot de l'isolation des données. Chaque entité a son propre jeu d'activités, rôles, outils et cartographie.

ColonneTypeRôle
idInteger PKIdentifiant
nameString(200)Nom de l'organisation
owner_idFK → users.idUtilisateur propriétaire
svg_contentTextSVG de cartographie sérialisé en base (persistance cloud)
optiqcarto_dataText (JSON)État complet de l'éditeur OptiqCarto (formes, connexions…)
vsdx_filenameString(255)Nom du dernier fichier Visio importé

Méthodes de classe clés : get_active(user_id) — entité active depuis la session ; set_active(entity_id) — change l'entité courante ; for_user(user_id) — toutes les entités d'un utilisateur.

Activities — activities

Activité métier extraite de la cartographie. Centrale dans l'application : toutes les compétences, tâches, outils et indicateurs lui sont rattachés.

ColonneTypeRôle
entity_idFK → entities.idEntité propriétaire
shape_idString(50)Identifiant de la forme SVG d'origine (unique par entité)
nameString(200)Nom de l'activité
is_resultBooleanL'activité est un résultat (type particulier dans la carto)
duration_minutesFloatDurée standard en minutes
delay_minutesFloatDélai standard en minutes

Relations : tasks, competencies, softskills, savoirs, savoir_faires, aptitudes, constraints — toutes en cascade delete.

Task — tasks

Tâche élémentaire d'une activité. Ordonnée (order). Reliée aux outils via many-to-many (task_tools) et aux rôles (task_roles).

Role — roles

Rôle/bande de la cartographie (ex: "Responsable", "Opérateur"). Peut avoir un plan d'onboarding (onboarding_plan). Relié aux activités via activity_roles (avec statut).

Tool — tools

Outil utilisé dans les activités/tâches. Peut avoir un fichier attaché (file_path → file_blobs). Unique par nom dans une entité.

Link — links

Connexion entre deux nœuds de la cartographie. Un lien relie soit deux activités, soit une activité et une donnée (Data). Les propriétés source_id/target_id abstraient cette dualité.

Performance — performances

Indicateur de performance attaché à un lien spécifique de la cartographie (unique par lien). Modèle léger : juste un nom et une description.

PerformancePersonnalisee — performance_personnalisee

Évaluation personnalisée d'un utilisateur sur une activité. Contient un champ content libre (JSON ou texte) et un statut de validation.

CompetencyEvaluation — competency_evaluation

Note d'un utilisateur sur un item de compétence d'une activité. L'item peut être n'importe quel type (item_type : competency, softskill, savoir, savoir_faire, aptitude). Supporte plusieurs sessions d'évaluation (eval_number).

TimeAnalysis — time_analysis

Saisie de temps pour une activité, tâche, rôle ou utilisateur. La récurrence (journalier, hebdo, mensuel, annuel) est utilisée pour calculer le temps annuel via annual_time.

Propriétés calculées : recurrence_factor (220j/42sem/10.5m/1an), annual_time = durée × facteur × fréquence × nb_people, delay_gap et delay_ratio pour l'analyse des écarts.

FileBlob — file_blobs

Stockage binaire des fichiers attachés aux outils et contraintes. Utilise LargeBinary (BYTEA en PostgreSQL, BLOB en SQLite) pour la persistance cloud.

RecentEvent — recent_events

Journal d'activité automatique alimenté par des SQLAlchemy event listeners sur les modèles Activities, Task, Role et Tool. Chaque création/modification/suppression insère un événement avec un label, une icône Font Awesome et un détail JSON (avant/après).

TaskLinkAssignment — task_link_assignments

Associe un lien de cartographie à une tâche avec une direction (incoming ou outgoing). Permet de savoir quels flux entrent/sortent d'une tâche précise.

Journal d'activité automatique

Le fichier models.py déclare des SQLAlchemy event listeners sur les modèles principaux. Ces listeners s'exécutent dans la même transaction que l'opération déclenchante et insèrent une ligne dans recent_events sans lever d'exception (les erreurs sont ignorées silencieusement pour ne pas bloquer les transactions métier).

Le pattern before_update + after_update est utilisé pour capturer le diff avant/après via sqlalchemy.inspect(target).attrs[field].history.
