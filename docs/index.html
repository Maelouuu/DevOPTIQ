Architecture Documenté

DevOPTIQ est une application web Flask déployée sur Google Cloud Run. Elle suit le pattern Application Factory : la fonction create_app() dans Code/app.py configure l'ensemble de l'application puis enregistre tous les blueprints. Chaque domaine fonctionnel est isolé dans son propre blueprint.

Vue d'ensemble de l'infrastructure DevOPTIQ

Flux de démarrage

Au démarrage du conteneur Cloud Run, create_app() exécute dans l'ordre :

  1. Configuration DB (PostgreSQL si DATABASE_URL présent, sinon SQLite local)
  2. Configuration Flask-Mail (SMTP Gmail)
  3. Enregistrement des 31 blueprints
  4. db.create_all() + ajout colonnes manquantes via ALTER TABLE + marquage Alembic
  5. Réinitialisation du pool de connexions DB avant d'accepter les requêtes HTTP

Gestion des fichiers sur Cloud Run

Cloud Run utilise un filesystem éphémère. DevOPTIQ contourne ce problème en persistant les données critiques en base : Entity.svg_content (SVG cartographie), Entity.optiqcarto_data (état JSON éditeur), FileBlob (pièces jointes).

Modèles de données Documenté

Fichier source : Code/models/models.py

Tous les modèles disposent d'un filtre for_active_entity() retournant une query filtrée sur l'entité active de la session. Cela garantit l'isolation des données entre organisations.

Schéma entité-relation principal. Toutes les données appartiennent à une Entity.

Description détaillée

Entity — entities

Tenant root de l'application. Toutes les données (activités, rôles, outils…) lui appartiennent via entity_id. L'entité active est stockée dans session['active_entity_id'] (pas en base).

Méthodes de classe : get_active(), get_active_id(), set_active(), for_user().

Activities — activities

Activité métier. shape_id la relie à une forme SVG dans OptiqCarto. Contrainte unique (entity_id, shape_id). Possède Tasks, Competencies, Softskills, Savoirs, SavoirFaires, Aptitudes, Constraints.

Role — roles

Correspond aux bandes horizontales de l'éditeur OptiqCarto. Unique par nom dans l'entité. onboarding_plan stocke la fiche de poste.

Link — links

Connexion directionnelle entre deux activités. Types : déclenchante (T-link) et nourrissante (N-link). Propriétés calculées source_id / target_id.

Task — tasks

Tâche ordonnée d'une activité. Reliée aux Tool via task_tools (many-to-many).

Champs : email (unique), password (hashé), first_name, last_name, status, manager_id (auto-référence hiérarchie). Relié aux rôles via UserRole et évaluations via CompetencyEvaluation.

Calcule annual_time = duration × recurrence_factor × frequency × nb_people. Facteurs : journalier=220, hebdo=42, mensuel=10.5, annuel=1.

SQLAlchemy event listeners enregistrent automatiquement dans recent_events toute création/modification/suppression des modèles Activities, Task, Role, Tool. _capture_changes() compare l'historique SQLAlchemy avant/après pour ne logger que les champs modifiés.

Cartographie — OptiqCarto Documenté

Éditeur SVG interactif JS vanilla. Deux repos synchronisés : DevOPTIQ/static/optiqcarto/ et standalone OptiqCarto/. Toujours synchroniser les deux après modification.

Deux composants

  • Éditeur OptiqCarto (/cartography/editor) — création formes/bandes/connexions, import VSDX, sauvegarde en base.
  • Carte des Activités (/activities/map) — gestion multi-entités, affichage SVG statique, wizard upload SVG+VSDX.

Format JSON (optiqcarto_data)

{
  "shapes": [
    { "id": "s1", "type": "process", "label": "Facturation",
      "x": 120, "y": 50, "w": 160, "h": 60 }
  ],
  "bands": [
    { "id": "b1", "label": "Direction", "height": 180 }
  ],
  "connections": [
    { "id": "c1", "fromId": "s1", "toId": "s2", "label": "Bon de commande" }
  ]
}

Types de shapes : process (activité normale), start-end (début/fin), special (résultat, is_result=True), decision (ignoré pour la sync DB).

Synchronisation carto → base (_sync_carto_to_db)

Appelée à chaque POST /cartography/api/save :

  1. Activités : upsert de chaque shape vers Activities. Shapes supprimées → suppression cascade.
  2. Rôles : upsert de chaque bande vers Role. Bandes disparues → suppression.
  3. Associations : recalcule l'appartenance activité-rôle par position Y de la shape dans les bandes.
  4. Liens : supprime les links activity→activity existants, recrée depuis les connexions du diagramme.

Persistance SVG

Double stockage pour gérer le filesystem éphémère Cloud Run :

  • Disque : static/entities/entity_{id}/carto.svg
  • Base : Entity.svg_content (texte complet). check_svg_exists() restaure depuis la DB si le disque est vide.

Gestion multi-entités

L'entité active est dans session['active_entity_id']. get_active_entity() : session → is_active=True DB → dernière entité du user.

GET /activities/api/cross_carto_matches — activités partagées entre l'entité active et les autres entités du même utilisateur.

Performance Documenté

Blueprint : performance_bp — fichier : Code/routes/performance.py, préfixe /performance.

Gère les indicateurs de performance attachés aux connexions entre activités (Link). Un indicateur porte un nom et une description ; il qualifie la valeur produite par une activité vers la suivante.

Modèle de donnée

Performance → link_id (FK vers Link) — un lien peut avoir zéro ou plusieurs performances.

Rendu fragmentaire (server-side HTML)

Chaque endpoint retourne un fragment HTML autonome rechargé par AJAX dans la fiche activité. La fonction _render_fragment(link_id) construit ce fragment : s'il n'existe aucune performance pour le lien, il affiche un message "Aucune performance générale" avec un bouton d'ajout.

Chatbot IA — Assistant OPTIQ Documenté

Blueprint : chatbot_bp — fichier : Code/routes/chatbot.py, préfixe /api/chatbot.

Interface conversationnelle propulsée par OpenAI GPT (gpt-4o-mini par défaut, configurable via OPENAI_CHATBOT_MODEL). Elle guide l'utilisateur dans la définition ou la révision des tâches d'une activité en appliquant la méthodologie OPTIQ.

Deux modes

ModeDéclencheurComportement
creerActivité sans tâchesDécouverte ouverte : une question à la fois, construction itérative de la liste
ameliorerActivité avec tâches existantesAnalyse critique : identifie surdétail, hors-périmètre, tâches manquantes

Règles OPTIQ injectées dans le prompt système

  • 5–8 tâches maximum (3–4 pour activités simples) — regroupement si excès
  • Pas de procédure ("cliquer sur…") — reformuler en tâche ("Saisir la demande")
  • Un verbe par tâche ; chaîne logique entrée → tâche → sortie
  • Protocole "Ça dépend" : condition + tronc principal + ≤ 2 branches (≤ 3 tâches chacune)

Contexte injecté par activité

Injection en base (POST /api/chatbot/inject)

Une fois les tâches validées par l'utilisateur dans l'UI, l'endpoint crée :

  • Task avec ordre et activity_id
  • Tool get-or-create (scope entité, insensible à la casse)
  • Data + Link si un outgoing_link est défini
La conversation est stateless côté serveur : l'historique complet (messages user + assistant) est renvoyé par le client à chaque appel à POST /api/chatbot/chat.

Authentification Documenté

Blueprint : auth_bp — fichier : Code/routes/connexion_routes.py

Authentification email/mot de passe, sessions Flask côté serveur uniquement (pas de JWT).

Flux de connexion

  1. POST /login — vérifie l'email, valide le hash avec check_password_hash()
  2. Si valide : stocke session['user_email'] et session['user_id'], redirige vers /activities/map
  3. Si invalide : flash() d'erreur, retour au formulaire
RouteMéthodeDescription
/loginGET, POSTFormulaire de connexion
/logoutGETEfface user_email, user_id, active_entity_id de la session
/auth/current_user_infoGETInfos utilisateur connecté en JSON (id, prénom, nom, rôles, manager)

Sécurité

  • Mots de passe : werkzeug.security.generate_password_hash (pbkdf2:sha256)
  • Clé session : variable d'environnement SECRET_KEY
  • Reset password : blueprint auth_password_bp (routes_password.py)

Patterns de contrôle d'accès

# Pattern 1 — vérification user_id
user_id = session.get('user_id')
if not user_id:
    return redirect(url_for('auth.login'))

# Pattern 2 — vérification email (cartography_editor)
def _require_auth():
    return bool(session.get("user_email"))
Pas de décorateur @login_required centralisé — chaque blueprint vérifie la session indépendamment.

Gestion du temps À documenter

À compléter : saisie des temps, analyse, projections. Modèles : TimeAnalysis, TimeProject, TimeProjectLine, TimeRoleAnalysis, TimeWeakness.

Activités — Fiche & Liste Documenté

Le module Activités est la pièce centrale de DevOPTIQ. Il représente chaque processus métier de l'entité active, avec ses tâches, connexions entrantes/sortantes, rôles, compétences et indicateurs. Le code est découpé en plusieurs fichiers spécialisés qui s'agrègent dans un seul blueprint.

Architecture modulaire

Le fichier activities.py est un agrégateur pur : il n'expose aucune route directement, il importe les sous-modules pour les enregistrer sur le blueprint commun activities_bp.

Page liste — GET /activities/view

Défini dans activities_view.py. Charge toutes les activités de l'entité active (Activities.for_active_entity()) et construit pour chacune un dictionnaire riche transmis au template display_list.html.

Import IA — Excel vers DB Documenté

Le module import_full.py (blueprint import_full_bp, préfixe /api/import-full) permet d'importer en masse les activités, tâches, outils, rôles et compétences depuis un fichier Excel fourni par le client. Le processus se déroule en deux étapes : analyse puis injection.

Flux global

Étape 1 (/analyze) : parsing + matching, retourne JSON. Étape 2 (/inject) : l'utilisateur valide les correspondances dans l'UI, puis déclenche l'écriture en base.

Format Excel attendu

Le parseur détecte automatiquement la feuille principale et identifie les colonnes par leurs en-têtes. Les cellules fusionnées sont propagées vers le bas pour reconstituer les groupes.

Colonne détectéeMots-clés reconnusRôle
activitysemi finish, semi-finish, activityNom de l'activité (groupement)
departmentdepartment, deptDépartement / entité
guarantorguarantor, garantRôle garant de l'activité
tasktask, tâcheTâche à créer
tooltool, outilOutil(s) séparés par virgule ou point-virgule
doerdoer, executorRôle exécutant (lié à la tâche)

Algorithme de matching (_algorithmic_match())

  1. Correspondance exacte (casse ignorée) — score 1.0
  2. Inclusion — l'un contient l'autre — score 0.88
  3. Fuzzy via difflib.SequenceMatcher — score entre 0 et 1
Seuil 0.90 : activité placée dans "Mappée" (confiance haute, import direct).
0.75–0.89 : placée dans "À résoudre" avec raison "Probable".
0.60–0.74 : placée dans "À résoudre" avec raison "Incertaine".
< 0.60 : aucune correspondance fiable — 3 suggestions triées par score sont proposées.

Enrichissement OpenAI (optionnel)

Si OPENAI_API_KEY est défini et qu'il reste des groupes non-matchés, _try_openai_enrich() appelle GPT (défaut gpt-4o-mini) pour tenter une résolution sémantique. Toute erreur (quota, réseau, JSON malformé) est ignorée silencieusement.

Injection en base (POST /api/import-full/inject)

  • Garant : recherché ou créé dans roles, lié à l'activité avec statut 'garant'.
  • Tâches : créées seulement si leur nom n'existe pas déjà (déduplication insensible à la casse).
  • Outils : get-or-create dans tools (scope entité), liés aux tâches.
  • Doer / Approbateur : rôles créés ou retrouvés, liés via task_roles.
  • Compétences : ajoutées à l'activité si la description n'existe pas encore.

  • JS : vanilla uniquement. $() = document.querySelector(). Un fichier par domaine dans static/js/.
